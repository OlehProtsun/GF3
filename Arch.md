# Project Architecture (AS-IS)

## 1. Executive Summary
- **Inferred product purpose**: desktop workforce/schedule management system for employees, shops, containers, availability groups, and monthly schedule generation/export (SQLite-backed).
- **Architecture style**: classic 3-layer layered architecture with MVVM presentation: **Presentation (WPFApp)** -> **BBL/BLL (BusinessLogicLayer)** -> **DAL (DataAccessLayer)**.
- **Core stack**: .NET 10, WPF, EF Core + SQLite, Microsoft.Extensions.Hosting + DI, ClosedXML export, CommunityToolkit.Mvvm package (plus custom MVVM primitives).
- **Current state after cleanup (AS-IS)**:
  - Foldering in WPF is materially improved (`MVVM`, `UI`, `Applications`, `View`, `ViewModel`, `Resources`, `DesignTime`).
  - BLL and DAL preserve repository/service layering and DI registration extensions.
  - Boundary is still partially mixed because WPF references DAL project and DAL models directly in many ViewModels/services (54 `.cs` files import `DataAccessLayer` namespaces).
  - Non-runtime BuildHost cleanups exist in project files (`BuildHost-net472`/`BuildHost-netcore` content removals), suggesting package-content pruning rather than runtime dependencies.
## 2. Solution and Project Structure
- **Solution file**: `GF3.sln` with 3 active projects: `DataAccessLayer`, `BusinessLogicLayer`, `WPFApp`.
- **Project roles**:
  - `DataAccessLayer` (**DAL**): EF Core entities, DbContext, repositories, migrations, SQLite provider setup.
  - `BusinessLogicLayer` (**BBL/BLL**): service abstractions + implementations, schedule generator, availability parsing/validation, DI extension.
  - `WPFApp` (**Presentation**): WPF views, ViewModels, converters, app-level UI/application services, resources/templates, startup/DI composition root.

```text
GF3/
├─ GF3.sln
├─ DataAccessLayer/
│  ├─ Models/ (entities, enums, DbContext)
│  ├─ Repositories/ (generic + feature repositories + abstractions)
│  └─ Migrations/ (historical EF migrations)
├─ BusinessLogicLayer/
│  ├─ Services/ (abstractions + implementations)
│  ├─ Generators/ (schedule generation contract/implementation)
│  ├─ Availability/ (parser/validator/payload builder)
│  └─ Common/ (ValidationException)
└─ WPFApp/
   ├─ App.xaml / App.xaml.cs (startup + DI root)
   ├─ View/ + ViewModel/ (feature UI and MVVM state)
   ├─ MVVM/ (base classes, commands, validation, threading)
   ├─ UI/ (dialogs, hotkeys, matrix UI helpers, color helpers)
   ├─ Applications/ (diagnostics, export, matrix engines, notifications)
   ├─ Converters/ + DesignTime/
   └─ Resources/ (styles, templates, control dictionaries, Excel templates)
```
## 3. Layered Architecture Overview
### 3.1 Presentation Layer (WPFApp)
- Responsibilities: user interaction, navigation, input editing/validation feedback, matrix visualization, dialogs, local app utilities (logging/admin/export wrappers), and orchestration of BLL services.
- Allowed dependencies (intended): BLL contracts/services + presentation infrastructure + DTO/view state.
- Actual dependencies (AS-IS): WPF references both BLL and DAL projects; many ViewModels/services use DAL models directly.
- Internal structure:
  - `MVVM/*`: `ObservableObject`, `ViewModelBase`, `RelayCommand`, `AsyncRelayCommand`, validation rules, debounced UI action helper.
  - `View/*`, `ViewModel/*`: feature modules (`Home`, `Employee`, `Shop`, `Availability`, `Container`, `Database`, `Dialogs`, `Information`, `Main`).
  - `UI/*`: dialog services and UI-specific helpers (hotkeys, color conversion, schedule column building).
  - `Applications/*`: non-visual app services (export, diagnostics, preview/matrix engines, database-change notifications, config provider).
  - `Resources/*`: global styles/themes/templates and Excel binary templates copied to output.
- DI composition/startup flow: Host builder in `App` creates service container, registers DAL/BLL + presentation services + VM/View types, starts host, resolves and shows `MainWindow`, stops host on exit.
### 3.2 BBL/BLL (Business Logic Layer)
- Responsibilities: CRUD orchestration on repositories, normalization/validation, aggregate schedule save with details, availability payload parsing/validation, and generation logic.
- Main structure: abstraction interfaces + concrete services, generic base service, standalone utilities (`Availability*`), and generator (`IScheduleGenerator`/`ScheduleGenerator`).
- Interaction model: BLL depends on DAL repository abstractions and DAL models; Presentation consumes BLL services.
### 3.3 DAL (Data Access Layer)
- Responsibilities: persistence model, EF configuration/constraints/indexes, repository query logic, migration history, and SQLite connectivity.
- Core artifacts: `AppDbContext`, `AppDbContextFactory`, repository abstractions/implementations, entity models (`*Model`), enums, migrations.
- SQLite infrastructure: connection string passed from WPF `DatabasePathProvider`; design-time factory falls back to `%LocalAppData%/GF3/SQLite.db`.
## 4. Dependency Flow and Boundaries
- **Intended flow**: `Presentation -> BLL -> DAL` is structurally encoded by project references (`BLL -> DAL`, `WPF -> BLL`).
- **AS-IS exception**: `WPFApp` also directly references `DataAccessLayer` project and imports DAL model namespaces widely (54 `.cs` files).
- **Boundary leaks found**:
  - ViewModels and WPF validation rules manipulate DAL entity models directly (tight compile-time coupling of UI to persistence schema).
  - Some application/UI services in WPF take DAL models as inputs/outputs (matrix/export/preview pipelines).

```text
Current dependency graph (AS-IS)

WPF Views/XAML
    ↓
WPF ViewModels / UI / Applications
    ├──> BusinessLogicLayer.Services (intended)
    └──> DataAccessLayer.Models (leak)
                ↓
BusinessLogicLayer.Services/Generators
                ↓
DataAccessLayer.Repositories
                ↓
AppDbContext (EF Core SQLite)
```

## 12. Known Architectural Risks and Improvement Opportunities
> Keep architecture concept unchanged (DAL/BBL/Presentation); focus is boundary hardening and cleanup.
- **Issue**: Presentation directly uses DAL models extensively
  - Where: WPF ViewModels, validation rules, matrix/export/preview services.
  - Why it matters: UI becomes tightly coupled to EF entity schema and DB evolution.
  - Recommended direction: Introduce BLL-facing DTO/view contracts incrementally while keeping 3-layer model..
- **Issue**: Large partial ViewModel surfaces
  - Where: `ContainerViewModel*`, `ContainerScheduleEditViewModel*`, `AvailabilityViewModel*`.
  - Why it matters: High cognitive load and hidden coupling between partial fragments.
  - Recommended direction: Continue partitioning by explicit sub-services and keep partials thin orchestrators..
- **Issue**: Mixed concerns in WPF application services
  - Where: `Applications/*` contains domain-ish matrix logic and UI-adjacent logic.
  - Why it matters: Harder test boundaries and ownership clarity.
  - Recommended direction: Define clear sub-namespaces for pure business helpers vs UI-application helpers..
- **Issue**: BLL thin-wrapper services
  - Where: Several services mainly proxy repositories.
  - Why it matters: Limited domain encapsulation despite BLL layer.
  - Recommended direction: Add normalization/validation/use-case orchestration in services where rules exist..
- **Issue**: Potential duplicated validation/parsing paths
  - Where: Validation in WPF + BLL + availability parsers.
  - Why it matters: Risk of drift between UI and business truth.
  - Recommended direction: Centralize canonical parsing/validation in BLL and keep WPF checks for UX only..
## 13. Recommended Target Organization (Within Current 3-Layer Architecture)
- **Presentation (WPFApp)**: keep `View`/`ViewModel` by feature; keep `MVVM`, `UI`, `Applications`, `Resources`; gradually move DAL model exposure behind BLL-oriented contracts/DTOs; maintain composition root in `App.xaml.cs`.
- **BBL/BLL**: keep service abstractions + concrete use-case services; expand true business policies in BLL (normalization/validation/generation), and keep parser/validator helpers here.
- **DAL**: keep EF entities/repositories/migrations as persistence details; continue strict query encapsulation in repositories and explicit DB constraints in `AppDbContext`.
- **Incremental next steps**:
  1. Introduce feature DTOs for one feature at a time (Employee/Shop first).
  2. Keep repository signatures DAL-only, but map to DTOs in BLL services.
  3. Refactor heavy ViewModels by extracting orchestration helpers into `WPFApp/Applications/*` or BLL use-cases where domain-centric.
  4. Add dependency checks in CI (simple Roslyn/analyzer/script) to prevent new direct WPF->DAL leakage growth.
## 14. Glossary and Naming Notes
- **Availability matrix**: tabular per-day availability input/preview for employees (codes like ANY/NONE/intervals).
- **Schedule matrix**: tabular day/employee assignment grid with slot/status/cell-style overlays and totals.
- **Bind(s)**: key/value bindings used for hotkeys or availability shortcuts (`BindModel`, bind services).
- **Preview**: computed visualization (non-persisted view model artifacts) before save/generate/export flows.
- **Container**: grouping entity that owns schedules and appears central in schedule workflows.
- Naming notes (AS-IS inconsistencies):
  - `Applications` folder name is plural; many projects commonly use `Application` or `Services`.
  - `BusinessLogicLayer` uses both BBL/BLL terminology in docs/comments.
  - Some migration names use inconsistent casing (`addNoteToSchedule`, `databaseFix`, `cellPaintAdded`).
  - Recommendation: standardize PascalCase for migration class names and choose one term (`BLL`) consistently.
